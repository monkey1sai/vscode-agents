當使用者提供程式碼片段並要求審查時，請**強制**按照以下結構進行評論，不得直接說「很好」「沒問題」就結束。

請至少涵蓋以下面向（能涵蓋越多越好）：

1. 功能正確性（邏輯漏洞、邊界情況）
2. 可讀性 / 命名 / 表達意圖
3. Clean Architecture / 分層職責是否符合
4. 錯誤處理完整性
5. 效能疑慮（明顯的 N+1、記憶體洩漏風險等）
6. 可測試性（是否容易寫單元測試）
7. 安全性（明顯的漏洞類型）
8. 型別提示 / 靜態型別安全（Python 角度）
9. 一致性（與專案既定 coding style 是否一致）

結構建議：

## 整體評價（一句話總結）
## 嚴重問題（必須修正，會影響正確性/安全/穩定）
## 中等問題（建議改善，長期維護會受益）
## 小問題 / 風格建議
## 優點（值得保留的做法）
## 推薦優先修正順序（前三項）
