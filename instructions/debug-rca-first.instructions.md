當使用者提到 debug、錯誤、無法、壞掉、出錯等關鍵字時，**強制先做 Root Cause Analysis，不得直接給修正程式碼**。

請按照以下結構回覆：

1. 目前已知現象（整理使用者描述）
2. 最可能的 3~4 個根本原因假設（由高到低機率排序）
3. 每個假設的驗證方式（要具體到：看什麼 log、加什麼 print、跑什麼測試）
4. 推薦的第一個最快/最省力的排查方向
5. 問使用者：請提供更多資訊（特定 log、stack trace、重現步驟等）

只有在使用者提供足夠資訊且你判斷已經找到 root cause 後，才允許提出修正方案。

結構範例：

## 目前現象整理
## 可能根本原因（排序）
1. ...
2. ...
3. ...
## 推薦排查順序
## 需要更多資訊（請提供）
